# gold-bot (Bybit / BTCUSDT) — заметки для Claude

Папка называется `gold-bot` по историческим причинам — проект начинался с
торговли золотом через MetaTrader5/XM, но **10.07.2026 полностью переведён на
Bybit (BTCUSDT perpetual)**. MT5 больше не используется вообще. Старый код и
результаты бэктеста по золоту/EURUSD удалены из рабочего дерева, остались
только в git-истории этой ветки — не пытайся их "восстановить" или опираться
на них как на актуальные, если явно не попросят.

## Кто пользователь

Али, начинающий алготрейдер, работает на Windows, русский — основной язык
общения. Объясняй решения в коде простыми словами, он учится. Отвечай по-русски.

## Жёсткие правила проекта (не нарушать)

1. **Никогда не подключай реальный счёт без явного письменного подтверждения
   Али в чате.** `config.py -> IS_DEMO_ONLY = True` — защитная заглушка,
   менять только по прямой просьбе с явным пониманием риска.
2. **Не переходим на следующий этап, пока текущий не закрыт.** Порядок:
   0-подключение -> 1-данные -> 2-исследование -> 3-бэктест -> 4-демо -> 5-live.
3. **Все риск-правила — в `risk_gate.py`, не в стратегии и не в `config.py`.**
   И бэктест, и (будущий) живой бот обязаны считать размер позиции ТОЛЬКО через
   `risk_gate.validate_order()` / `compute_position_size()`. Стратегия не
   должна и не может обходить эти проверки.
4. **Стоп-лосс обязателен у каждого ордера** — без него `risk_gate` бросает
   `OrderRejected`, ордер не отправляется. Это проверяется в коде, а не "на бумаге".
5. **Бэктест минимум 6 месяцев** истории, прежде чем финализировать параметры
   стратегии — не подгонять параметры под красивую кривую доходности
   (оверфиттинг). Если результат плохой — говорить честно, отрицательный
   результат тоже результат.
6. Перед тем как объявить что-то готовым — проверять на синтетических данных
   локально (эта среда без доступа к интернету наружу — Bybit API заблокирован
   политикой сети песочницы, 403 от прокси; реальные данные и прогон бэктеста
   на них может сделать только Али на своей машине).

## Стек

- Python 3.11+, Windows (у Али)
- **pybit** (официальный SDK Bybit) — НЕ ccxt. Причина выбора — см. README.md
  ("Почему pybit, а не ccxt"): нужна честная поддержка Bybit Demo Trading.
- pandas, numpy, matplotlib, pyarrow, python-dotenv
- Секреты — только через `.env` (шаблон `.env.example`), никогда не хардкодить
  в код. `.env` в `.gitignore`.

## Архитектура

- `risk_gate.py` — единая точка риск-проверки (константы: `STARTING_BALANCE_USDT`,
  `RISK_PER_TRADE_PCT`, `MAX_EFFECTIVE_LEVERAGE`, `DAILY_LOSS_LIMIT_PCT`,
  `MAX_LOSING_TRADES_PER_DAY`, `MIN_ORDER_USDT`, `TAKER_FEE_PCT`,
  `ASSUMED_FUNDING_RATE_PER_8H`). Класс `DailyState` — состояние дневных
  лимитов, сбрасывается по календарным суткам UTC.
- `backtest/engine.py` — цикл бэктеста: держит позицию, пока не сработает
  стоп ИЛИ сигнал стратегии на выход (НЕ окно, как было у золота — крипта 24/7).
  Funding считается по факту пересечения 00:00/08:00/16:00 UTC во время
  удержания позиции.
- `backtest/strategies.py` — EMA(9/21) trend-following, стоп 2×ATR(14).
  Параметры — первая гипотеза, не подогнанные числа.
- `bot/bybit_client.py` — обёртка pybit; режим (testnet/demo/live) целиком из
  `.env`, `warn_if_live()` явно предупреждает при обнаружении live-конфига.
- `bot/fetch_history_bybit.py` — публичный REST kline API, ключи НЕ нужны.
- `bot/run_live.py` — заглушка Этапа 4, ещё не реализована намеренно (см. правило 2).

## Известные ограничения на сегодня

- `ASSUMED_FUNDING_RATE_PER_8H` в `risk_gate.py` — приближение (0.01%/8ч),
  не измеренный факт. Стоит уточнить через реальную историю funding rate
  Bybit API перед тем как доверять бэктесту количественно.
- Минимальный ордер Bybit (`MIN_ORDER_USDT = 5.0`) — примерное значение,
  для конкретного инструмента (BTCUSDT) стоит сверить точный лимит через
  `session.get_instruments_info()`.
- Новостной календарь (`config.NEWS_DATES_UTC`) пуст — TODO на будущее, для
  крипты менее критично, чем было для золота/форекс, но не забыть при желании.
